1. Project Background

The Library Management System is a relational database project designed to manage book inventories, reader memberships, staff operations, and book transactions like issuing and returning. It reflects a real-world library environment, ensuring data accuracy, simplified access, and secure operations through normalized tables, foreign key constraints, and structured SQL queries and views.

Key Table Purposes:

  • READERS Table:

    Purpose: Stores all personal information of registered library users.

    Columns: User_ID (Unique ID), Firstname, Lastname, Email, Phone_no, Address.

    Why: Enables accurate identification of each user and helps in managing borrow history.

  • BOOKS Table:

    Purpose: Stores detailed information about every book in the library.

    Columns: ISBN (Unique ID), AuthNo, Title, Edition, Category, Price, Publisher_id.

    Why: Tracks all book data and links books to publishers and reservations.

  • PUBLISHER Table:

    Purpose: Stores information about book publishers.

    Columns: Publisher_id (Unique ID), Name, YearOfPublication.

    Why: Maintains source details for each book and supports book classification.

  • STAFF Table:

    Purpose: Holds information about staff members managing the library.

    Columns: Staff_id (Unique ID), Name, LoginId.

    Why: Ensures authorized staff access and links actions (like issuing books) to responsible individuals.

  • AUTHENTICATIONSYSTEM Table:

    Purpose: Stores login credentials for staff.

    Columns: LoginId (Unique ID), Password.

    Why: Secures staff access to the system and links credentials to the Staff table.

  • RESERVERETURNDATE Table:

    Purpose: Records reservation and return dates of books by readers.

    Columns: User_ID, ISBN, ReserveDate, ReturnDate.

    Why: Tracks which books were borrowed by whom, when, and if returned.

  • REPORTS Table:

    Purpose: Records book issue and return actions handled by staff.

    Columns: Staff_id, Book_No, User_id, IssueReturn.

    Why: Helps generate reports of transactions and track staff activities.

2. Description of the Project

The Library Management System is a software tool that helps run the daily activities of a library. It is used to keep track of books, members, and book transactions like issuing and returning books. This system saves time for librarians and helps reduce mistakes that can happen with manual work. It uses a Database Management System (DBMS) to store and manage important data such as book details, member records, and issue/return history. With this system, it becomes easy to add new books, register members, and quickly search for available books.

The system has different features for both librarians and users. Librarians can manage the entire system, issue books, and check reports. Users can view available books and their own borrowing history. The system also helps in tracking due dates and calculating fines for late returns. Overall, this project helps make the library more organized and efficient by reducing paperwork, avoiding data loss, and making it easier to update and find information. It is a simple and practical project to understand how databases are used in real life.

6. Description of Tables

AuthenticationSystem

Purpose: Stores login credentials for staff.

Attributes:

  • LoginId: Unique identifier for login credentials.
  • Password: Encrypted or plain text password (ideally hashed).

Staff

Purpose: Holds information about library staff members.

Attributes:

  • Staff_id: Unique identifier for staff.
  • Name: Staff member’s full name.
  • LoginId: Foreign key linking to AuthenticationSystem for login.

Relationships:

  • Logs in via AuthenticationSystem.
  • Manages transactions in the Reports table.

Readers

Purpose: Contains detailed information about library users (readers).

Attributes:

  • User_ID: Unique ID for each reader.
  • Firstname, Lastname: Reader’s names.
  • Email: Reader’s email address.
  • Phone_no: Contact number.
  • Address: Residential address.
  • Name: Full name or display name (may duplicate first/last name).

Relationships:

  • Keeps track of books through ReserveReturnDate.
  • Appears in Reports when books are issued/returned.

Publisher

Purpose: Represents publishing companies of books.

Attributes:

  • Publisher_id: Unique ID for each publisher.
  • Name: Publisher name.
  • Yearofpublication: Year when they published the book(s).

Relationships:

  • Publishes multiple Books.

Books

Purpose: Contains book details available in the library.

Attributes:

  • ISBN: International Standard Book Number (unique).
  • AuthNo: Author ID or reference.
  • Title: Name of the book.
  • Edition: Book edition.
  • Category: Genre/category (e.g., Programming, Fiction).
  • Price: Price of the book.
  • Publisher_id: Foreign key linking to the Publisher table.

Relationships:

  • Maintained in ReserveReturnDate for tracking.
  • Managed in Reports for issuing/returning.
  • Linked to a Publisher.

ReserveReturnDate

Purpose: Tracks reservation and return activity between readers and books.

Attributes:

  • User_ID: Foreign key to Readers.
  • ISBN: Foreign key to Books.
  • ReserveDate: When the book was reserved.
  • ReturnDate: When the book was returned (null if not returned).
  • dueDate: The date it should be returned.

Primary Key: Composite key (User_ID, ISBN).

Reports

Purpose: Captures transaction logs of book issues/returns.

Attributes:

  • Reg_no: Unique transaction record number.
  • User_ID: Reader involved in the transaction.
  • Book_No: Book involved (same as ISBN).
  • IssueReturn: Status, either 'Issue' or 'Return'.
  • Staff_id: Staff who performed the operation.

Relationships: Connected to Readers, Books, and Staff.

7. Normalization of Tables

1. First Normal Form (1NF) – Eliminate repeating groups

Tables already satisfy 1NF based on the ER diagram:

  • Each cell has atomic (indivisible) values.
    • Attributes like Firstname, Lastname, Email, Price, etc., contain single, indivisible values.
  • Each row is unique (Primary Keys).
    • Entities like Readers, Books, Staff, Publisher, AuthenticationSystem, etc., all have a primary key (User_ID, ISBN, Staff_id, etc.) ensuring row uniqueness.

2. Second Normal Form (2NF) – Eliminate Partial Dependencies

Applies to tables with composite primary keys. We check if non-key attributes depend on the entire key, not part of it. Your schema satisfies 2NF because:

  • For instance:
    • In Readers, fields like Firstname, Lastname, Email, Phone_no, and Address are fully dependent on User_ID.
    • In Books, attributes such as Title, Category, Edition, and Price depend on the primary key ISBN.
    • In Reports, Book_No and IssueReturn are fully dependent on the primary key Reg_no.

Note: For ReserveReturnDate with composite key (User_ID, ISBN), attributes like ReserveDate, ReturnDate, DueDate depend on the whole key.

3. Third Normal Form (3NF) – Remove transitive dependencies

No non-prime attribute should depend on another non-prime attribute. It is already in 2NF. There are no transitive dependencies; no non-key attribute depends on another non-key attribute.

Examples:

  • In Books, attributes like Title, Category, Edition, and Price all depend directly on the primary key ISBN — not on each other. (Assuming Publisher_id is a foreign key and doesn't introduce transitive dependency for other book attributes).
  • In Reports, User_ID and Book_No are foreign keys, and IssueReturn is directly related to the transaction record — no indirect dependencies between non-key fields.

All non-key attributes are only dependent on primary keys, and nothing else.

9. SQL Queries (Subqueries, Aggregate Functions, Joins)

Joins

Get all books along with their publisher names:


  SELECT b.Title, b.Category, p.Name AS Publisher
  FROM Books b
  JOIN Publisher p ON b.Publisher_id = p.Publisher_id;

Get staff names and the titles of books they issued or returned:


  SELECT s.Name AS StaffName, rp.IssueReturn, b.Title AS BookTitle
  FROM Reports rp
  JOIN Staff s ON rp.Staff_id = s.Staff_id
  JOIN Books b ON rp.Book_No = b.ISBN;

Subqueries

Find names of readers who reserved books published by 'Pearson':


  SELECT Firstname, Lastname
  FROM Readers
  WHERE User_ID IN (
      SELECT rr.User_ID
      FROM ReserveReturnDate rr
      JOIN Books b ON rr.ISBN = b.ISBN
      JOIN Publisher p ON b.Publisher_id = p.Publisher_id
      WHERE p.Name = 'Pearson'
  );

Find book titles with price greater than the average book price:


  SELECT Title, Price
  FROM Books
  WHERE Price > (
      SELECT AVG(Price) FROM Books
  );

Aggregate Functions

Count how many books each publisher has:


  SELECT p.Name AS Publisher, COUNT(b.ISBN) AS TotalBooks
  FROM Publisher p
  JOIN Books b ON p.Publisher_id = b.Publisher_id
  GROUP BY p.Name;

Find the number of books reserved by each reader:


  SELECT r.User_ID, r.Firstname, r.Lastname, COUNT(rr.ISBN) AS ReservedCount
  FROM Readers r
  JOIN ReserveReturnDate rr ON r.User_ID = rr.User_ID
  GROUP BY r.User_ID, r.Firstname, r.Lastname;

10. Creation of Views

reader_book_view

Shows which reader has reserved which book and the reservation date.


  CREATE VIEW reader_book_view AS
  SELECT r.User_ID, r.Firstname, r.Lastname, b.Title, rr.ReserveDate
  FROM Readers r
  JOIN ReserveReturnDate rr ON r.User_ID = rr.User_ID
  JOIN Books b ON rr.ISBN = b.ISBN;

book_publisher_view

Lists books with their details and publisher name.


  CREATE VIEW book_publisher_view AS
  SELECT b.ISBN, b.Title, b.Category, b.Price, p.Name AS PublisherName
  FROM Books b
  JOIN Publisher p ON b.Publisher_id = p.Publisher_id;

staff_transaction_view

Displays staff members, the transactions (issue/return) they handled, the book title, and the user involved.


  CREATE VIEW staff_transaction_view AS
  SELECT s.Staff_id, s.Name AS StaffName, rp.IssueReturn, b.Title AS BookTitle, rp.User_id
  FROM Reports rp
  JOIN Staff s ON rp.Staff_id = s.Staff_id
  JOIN Books b ON rp.Book_No = b.ISBN;

reader_contact_view

Provides a quick lookup for reader's full name and contact information.


  CREATE VIEW reader_contact_view AS
  SELECT User_ID, Firstname || ' ' || Lastname AS FullName, Email, Phone_no
  FROM Readers;

publisher_book_count_view

Shows the total number of books associated with each publisher.


  CREATE VIEW publisher_book_count_view AS
  SELECT p.Publisher_id, p.Name AS PublisherName, COUNT(b.ISBN) AS TotalBooks
  FROM Publisher p
  LEFT JOIN Books b ON p.Publisher_id = b.Publisher_id -- Use LEFT JOIN to include publishers with no books
  GROUP BY p.Publisher_id, p.Name;

book_price_category_view

Categorizes books based on their price range.


  CREATE VIEW book_price_category_view AS
  SELECT
      ISBN, Title, Category, Price,
      CASE
          WHEN Price < 30 THEN 'Low'
          WHEN Price BETWEEN 30 AND 60 THEN 'Medium'
          ELSE 'High'
      END AS PriceRange
  FROM Books;
